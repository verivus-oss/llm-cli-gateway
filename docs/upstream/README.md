# Upstream provider tracking

This directory tracks the upstream CLIs the gateway shells out to (Claude Code,
Codex, Gemini/Antigravity, Grok, Mistral Vibe, Devin, and Cursor Agent) so
flag/output/session/permission drift is caught before it reaches a user.

## Single mechanical source of truth

`src/upstream-contracts.ts` (`UPSTREAM_CLI_CONTRACTS`) is the **only** place that
defines and enforces each provider's argv/env behaviour — flags, arities, enum
values, output modes, session/resume rules, permission modes, forbidden flags,
env contracts, positional limits. Enforcement happens through
`validateUpstreamCliArgs` / `validateUpstreamCliEnv`. Nothing in this directory
re-encodes those rules.

## Files

| Path                          | Role                                                                                                                                                                                              |
| ----------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `release-targets.md`          | Exact upstream CLI versions and artifact SHA-256 values from release validation and explicitly labelled pending contract probes. Neither is a dependency lock or minimum-supported-version claim. |
| `provider-sources.dag.toml`   | **Scanner input only.** Maps each canonical `CliType` to its changelog/release URLs and a scan policy. Never consulted for contract enforcement.                                                  |
| `snapshots/<cli>.json`        | Generated by `--write-snapshot` with live sources and/or installed CLI probes. Stores source fingerprints plus preserved help and command evidence for later drift checks.                        |
| `reports/YYYY-MM-DD-<cli>.md` | Generated by `npm run upstream:scan -- --write-report`. Human-readable advisory report.                                                                                                           |

`snapshots/` and `reports/` only contain generated artifacts; they are written
exclusively by `scripts/upstream-scan.mjs` under the `--write-snapshot` /
`--write-report` flags.

## Detection channels

The detector has two complementary legs (both advisory only):

1. **Web changelog pages.** The scanner records a source fingerprint for each
   tracked `sourceUrl` (from each provider's `upstreamMetadata`) and emits a
   `watched-category-changed` finding when its material content changes. Most
   sources use a content hash. GitHub release APIs use stable release fields
   instead of the mutable API envelope, and the Codex RSS feed uses selected
   CLI/watch-entry semantics. This works well for providers that publish
   detailed release notes (Claude, Codex, Gemini, Mistral).

2. **Installed binary help surface** (the improvement that makes the detector
   reliable for vendor / fast-moving CLIs): when you pass `--probe-installed`
   to either an offline or live scan (or use the MCP/CLI `upstream_contracts` / `contracts`
   surfaces with `probeInstalled: true`), the scanner separately runs
   `--version`, parses root `--help`, then probes only verified catalogued
   command paths. It extracts advertised root flags, root commands, and the
   fully validated root request surface's arities and closed enum choices. The
   catalogued subcommand surface remains name and flag-drift checked, and
   reports:
   - `extraFlags` / `installed-help-surface-drift`: flags the binary knows
     about that the contract in `src/upstream-contracts.ts` neither declares
     nor acknowledges.
   - `missingFlags` / `binary-missing-declared-flags`: the reverse
     (regressions). Contract flags marked `hiddenFromHelp` (real, accepted,
     but deliberately absent from the binary's `--help` — e.g. Claude's
     `--max-turns`) are excluded; if such a flag reappears in the help text
     the probe emits a stale-marker warning instead.
   - `acknowledgedExtraFlags`: installed-binary flags listed in the contract's
     `acknowledgedUpstreamFlags` (upstream-only surface the gateway
     deliberately never emits, such as long-form aliases and interactive-only
     switches).
     Filtered out of `extraFlags` so genuinely new upstream flags stand out;
     acknowledged flags that vanish from the help text produce a stale-entry
     warning. This list is probe-only and never loosens the argv allowlist.
   - Version drift against `PROVIDER_TARGET_VERSIONS`, including target
     mismatches that previously could look clean when only flag names matched.
   - Added or removed root commands. A declared command missing from root help
     is recorded as unavailable and never probed through fallback root help.
   - Incompatible emitted root-flag arity and parsed closed enum values where
     help provides enough structure. An intentionally narrower gateway form,
     such as always passing one value to an optional or variadic upstream flag,
     does not produce arity drift.
   - Hash and set diffs against the previous help surface snapshot, including
     canonical `helpSurface.discoveredFlags`, root commands, version target and
     result data, plus source-specific semantic fingerprints. GitHub release
     APIs retain release identity and body semantics. The broad Codex RSS feed
     retains only stable CLI-anchored or watched non-app/mobile entries, keyed
     by GUID and normalized content, so feed ordering, timestamps, and routine
     app/mobile posts do not create contract-drift findings.

   This is the primary reliable signal for Grok (vendor channel, high-level
   release notes page) and any future CLIs whose public changelogs lag the
   actual `--help` / behavior.

Both channels feed the same findings list and can be persisted together in
the per-CLI snapshot JSON when `--write-snapshot` is used.

## How the TOML and the TypeScript metadata stay in sync

The chosen model is **TOML-is-scanner-input + report-enriched-from-TS**:

1. `CliContract.upstreamMetadata` in `src/upstream-contracts.ts` carries the
   authoritative `sourceUrls` and `watchCategories` (plus `packageName`,
   `repo`, `installDocsUrl`, `releaseChannel`). The contract **report** is
   enriched from these fields.
2. `provider-sources.dag.toml` mirrors `source_urls` and `watch_categories`
   for the scanner's offline scan plan only.
3. `src/__tests__/upstream-sources.test.ts` asserts the two are identical for
   every provider, so editing one without the other fails CI.

The TOML is **never** loaded for mechanical contract enforcement — only the
scanner reads it. To change a source URL or watch category, edit
`src/upstream-contracts.ts` first, then mirror it into the TOML.

## Commands

```bash
# Blocking, offline, network-free gate (fixtures + report + TOML-sync).
npm run upstream:contracts

# Offline scanner summary (no network, no writes).
npm run upstream:scan

# Advisory live changelog check (manual; network required).
npm run upstream:scan -- --live --fail-on-critical

# Persist source hashes / write a report (manual).
npm run upstream:scan -- --live --write-snapshot --write-report

# Recommended for vendor / fast-moving CLIs: probe the installed binary's
# --help surface without requiring network access. Replace grok with any
# canonical CliType.
npm run upstream:scan -- --provider grok --probe-installed --fail-on-critical

# Persist source hashes / reports while also including installed help-surface data.
npm run upstream:scan -- --live --provider grok --probe-installed --write-snapshot --write-report
```

The default release gate is `npm run check && npm run upstream:contracts`.
Neither requires network access nor installed provider CLIs. Live scans are
advisory and run manually; `--probe-installed` is an opt-in local probe that
works in offline and live scan modes.

## Release target matrix

Every release must update `release-targets.md` with the exact upstream CLI
versions and SHA-256 values observed in the release validation environment.
Treat those rows as the concrete provider behavior baseline for agent guidance
and support triage. Do not describe a target/probed version or artifact hash as
a minimum, maximum, pinned, installed-for-users, or guaranteed-compatible
version unless a separate compatibility gate proves that claim.

The installed-binary probe compares its explicit `--version` result to
`PROVIDER_TARGET_VERSIONS`. Update that target, the contract/fixtures, and the
release-target evidence together after a confirmed upstream upgrade.
